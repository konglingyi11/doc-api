# 自定义 API Key 与 Base URL 文档站设计规格

## 1. 项目目标

构建一个纯静态中文文档网站，教用户如何在常见编程工具中自定义 `API Key` 与 `Base URL`，并将 `https://api.1010101.asia/` 作为首版示例中的默认 `Base URL`。站点首版优先服务希望尽快完成配置的用户，同时保留新手完成配置所需的最低必要解释。

首页负责快速分流与统一说明，工具页负责让用户在单页内完成“找到入口 → 完成配置 → 验证生效 → 基本排错”的闭环。

## 2. 目标用户

- 希望将常见 AI 编程工具接入自定义接口地址的普通用户
- 不熟悉 `API Key`、`Base URL`、Provider 概念的新手用户
- 希望快速复制配置、尽快完成接入的有经验开发者

### 2.1 用户优先级判断

首版内容组织优先保证有经验用户能最快完成配置，同时确保新手在不跳出站点的前提下也能理解完成配置所需的最小概念。

## 3. 设计原则

1. 以可执行为第一优先级，避免只讲概念不讲配置。
2. 以最短路径帮助用户完成配置：首页讲通用步骤，工具页直接给操作方法。
3. 先写动作，再写解释；先给闭环，再给补充说明。
4. 站点内容自包含，不依赖外部链接完成核心流程。
5. 图片资源全部下载到本地并随站点一起发布，不引用远程图片。
6. 首版优先保证结构稳定、内容完整、后续易扩展。
7. 页面风格保持干净、轻量、偏官方文档体验，不做复杂交互。
8. 模型名、Provider 名等易变信息非必要不写，必须写时才给出可用示例。

- 配置位置

- 配置示例
- 验证方法
- 常见报错
- 必要的平台差异注意事项

### 4.4 首版完成标准

首版不是“9 个工具页都上线”就算完成，而是 9 个工具页都必须达到可直接照做的完成态，不做占位页，不保留“后续补充”。

### 5.1 顶级结构

- 首页
- 通用说明

  - 什么是 Base URL
  - 通用配置原则
  - 如何验证配置是否生效
- 工具配置
  - Claude Code
  - Codex
  - OpenCode
  - Cline
  - Cherry Studio
  - Continue
  - Roo Code
  - Cursor
  - Windsurf
- 常见问题

### 5.2 首页职责

首页只做四件事：

1. 用最短篇幅解释用户真正需要修改的是 `API Key` 与 `Base URL`
2. 给出统一默认 `Base URL`：`https://api.1010101.asia/`
3. 给出 3 步快速开始
4. 提供按工具进入的清晰入口

首页不承担完整教学，不展开大段背景知识，只负责高效分流。

### 5.3 首页模块顺序

首页采用 VitePress 默认 home layout，模块顺序固定为：

1. Hero 区
2. 3 步快速开始
3. 工具入口卡片区
4. 通用说明摘要区
5. FAQ 预览区

### 5.4 首页信息层级

首页信息按操作优先级组织：

- 第一层：立即可抄的信息
  - 标题说明这是自定义 `API Key` 与 `Base URL` 配置文档
  - Hero 区直接展示默认 `Base URL`
  - 提供明确的开始按钮
- 第二层：最短行动路径
  - 3 步快速开始
- 第三层：按工具分流
  - 9 个工具卡片入口
- 第四层：补充说明
  - 通用说明摘要与 FAQ 预览

### 5.5 导航结构

顶部导航固定为：

- 首页
- 通用说明
- 工具配置
- 常见问题

顶部导航不直接展开全部 9 个工具，避免导航过长。具体工具切换交给首页卡片区与侧边栏。

### 5.6 侧边栏结构

侧边栏分组：

- 通用说明
  - 什么是 API Key
  - 什么是 Base URL
  - 通用配置原则
  - 如何验证配置是否生效
  - 常见问题
- 工具配置
  - Claude Code
  - Codex
  - OpenCode
  - Cline
  - Cherry Studio
  - Continue
  - Roo Code
  - Cursor
  - Windsurf

### 5.7 导航行为原则

- 首页是首次进入入口，不是唯一入口
- 用户从搜索结果或直链进入工具页时，也必须能独立完成配置
- 首页工具卡片顺序与侧边栏工具顺序尽量一致
- FAQ 是补充页，不替代工具页内部排错模块

## 6. 内容组织

### 6.1 首页内容策略

首页偏向概览与引导，不承载长篇细节。用户进入首页后，应能在几十秒内理解以下信息：

- 为什么需要配置 `API Key`
- 为什么有些工具需要自定义 `Base URL`
- 默认示例地址是什么
- 自己应该点击哪个工具页继续

### 6.2 3 步快速开始

首页的 3 步快速开始固定写成操作导向：

1. 准备自己的 `API Key`
2. 把工具里的接口地址改成 `https://api.1010101.asia/`
3. 进入对应工具页完成配置并验证

这部分文案只讲动作，不展开背景解释。

### 6.3 工具页固定模板

每个工具页采用统一结构：

1. 这是什么工具
   - 说明这是命令行工具、编辑器插件还是桌面应用
2. 配置前需要准备什么
   - `API Key`
   - 默认 `Base URL`：`https://api.1010101.asia/`
   - 必要时补充是否需要重启工具、重新打开会话等
3. 从哪里打开配置入口
4. 最短完整配置步骤
   - 从打开设置入口开始
   - 到填写、保存、重新加载或重启结束
5. 配置示例
   - 只给该工具真正需要的片段
6. 如何验证生效
   - 说明操作动作与成功现象
7. 常见报错与排查
8. 注意事项 / 平台差异
9. 关键截图

### 6.4 平台差异组织方式

每个工具页主体以通用步骤为主。只有确实影响操作结果的平台差异才放进“注意事项”，不按平台拆成独立页面。

### 6.5 配置示例粒度

每个工具页的配置示例都必须覆盖从“打开哪里”到“粘贴、保存、验证”的最短完整链路，而不是只给最终配置片段。

### 6.6 模型名与 Provider 名策略

- 非必要不写模型名或 Provider 名
- 某工具必须填写模型名或 Provider 名时，再给出可用示例
- 不把易变信息大面积写死，降低维护成本

### 6.7 内容粒度约束

每个工具页以“配置位置 + 完整步骤 + 配置示例 + 验证方法 + 常见报错”为最小闭环，不额外扩展大量背景知识。背景解释统一放在通用页，避免工具页重复冗长。

## 7. 视觉与交互设计

### 7.1 视觉风格

- 干净、轻量、偏官方文档风格
- 重点突出“清楚、快找、能复制”
- 首页允许使用卡片式入口提升可读性
- 支持浅色与暗色模式，但不做额外品牌化视觉系统

### 7.2 首页样式策略

首页使用 VitePress 默认 home layout，只做轻量定制，不做产品落地页式的大幅自定义。

Hero 区要求：

- 标题强调快速配置
- 副标题说明站点帮助用户修改 `API Key` 与 `Base URL`
- 默认 `Base URL` 直接可见
- 提供动作型主按钮，例如“开始配置”

### 7.3 图片策略

- 图片是辅助，不替代文字步骤与配置代码块
- 图形界面工具优先配关键截图：Cline、Cherry Studio、Continue、Roo Code、Cursor、Windsurf
- 命令行工具以文字步骤和代码块为主：Claude Code、Codex、OpenCode
- 每个工具页控制在 1 到 3 张关键截图
- 所有图片资源下载到本地后再引用
- 截图只服务于入口定位、关键字段确认、成功确认三个场景

### 7.4 交互边界

- 不做登录
- 不做在线测试接口
- 不做后端服务
- 不做复杂动画
- 首版只保留文档阅读所需的基础交互

## 8. 资料整理策略

1. 可参考公开资料核对当前配置方式。
2. 站内内容统一重写，保持术语和结构一致。
3. 站内不放外部参考链接。
4. 用户完成配置所需的信息，尽量全部在站内闭环提供。
5. 对界面频繁变化的工具，优先写“路径描述 + 字段名 + 关键词”，降低截图过期风险。

## 9. 报错与 FAQ 组织策略

### 9.1 工具页报错组织原则

每个工具页都要自带最常见的报错排查，FAQ 不替代工具页排错。

每条报错统一按以下结构组织：

- 现象
- 更可能的原因
- 最短排查动作
- 仍未解决时再看哪里

### 9.2 工具页应优先覆盖的报错类型

- `401 Unauthorized`
- `404 Not Found`
- 模型列表为空
- 连接超时
- 保存后未生效

所有报错都要给出可执行排查动作，而不只列错误名。

### 9.3 通用 FAQ 的边界

FAQ 只保留跨工具共性问题，例如：

- 为什么改完设置后没有生效
- `API Key` 和 `Base URL` 通常分别填在哪里
- 如何判断工具是否真的在走自定义接口
- 为什么有些工具还要求填模型名或 Provider
- 为什么不同平台看到的界面略有不同

## 10. 技术方案

### 10.1 框架选择

采用 VitePress 构建文档站。

选择原因：

- 适合技术文档
- 纯静态输出，部署简单
- 导航、侧边栏、代码块与搜索体验成熟
- 后续新增工具页维护成本低

### 10.2 实现策略

- 首页使用默认 home layout
- 导航与侧边栏通过 `.vitepress/config.ts` 统一维护
- 主题基于默认主题，只做必要的轻量定制
- 保留默认暗色模式支持
- 代码块、目录、搜索优先使用 VitePress 原生能力

### 10.3 部署形式

- 输出为纯静态站点
- 便于部署到 GitHub Pages、Vercel、Nginx 等静态托管环境
- 首版不依赖任何后端基础设施

### 10.4 搜索策略

首版采用 VitePress 自带的本地搜索能力或等价轻量配置，不引入单独搜索服务。

## 13. 页面行为要求

### 13.1 首页

首页必须让用户快速完成以下动作：

- 理解通用概念
- 获得默认 `Base URL`
- 获得 3 步快速开始
- 进入目标工具页

首页首屏必须直接可见默认 `Base URL`，且用户无需跳页即可看到 3 步快速开始。

### 13.2 工具页

工具页必须让用户在单页内完成以下闭环：

- 找到配置入口
- 复制或填写可用示例
- 保存配置
- 验证配置成功
- 遇到错误时完成基本排查

### 13.3 搜索与跳读

- 用户通过搜索或直链进入任意工具页时，也应能独立完成配置
- 本地搜索至少能覆盖工具名、`API Key`、`Base URL`、常见错误关键词

## 14. 非目标

首版不包含以下内容：

- 多语言支持
- 账号系统
- 用户评论或反馈系统
- 在线 API 联通性检测
- 自动抓取官方文档并同步
- 高度定制化可视化首页动画
- 依赖外部服务的首版关键能力

## 15. 验收标准

当以下条件全部满足时，首版设计视为达标：

1. 网站可作为纯静态站点部署。
2. 首页能清晰解释 `API Key` 与 `Base URL` 的基本概念。
3. 首页首屏直接可见默认 `Base URL`，且包含 3 步快速开始。
4. 已覆盖 9 个指定工具的独立页面。
5. 每个工具页都包含配置位置、最短完整配置步骤、配置示例、验证方法、常见报错四个核心区块。
6. 任一工具页都能在单页内完成“入口定位 → 配置 → 保存 → 验证 → 基本排错”的闭环。
7. 所有工具页使用统一模板，章节顺序基本一致。
8. CLI 工具页不依赖截图才能完成操作，GUI 工具页的截图只做辅助定位。
9. 所有站内图片均为本地资源引用。
10. 用户不依赖外部链接即可完成主要配置流程。
11. 页面中不出现“待补充”“后续完善”“参考外部文档”等占位表达。
12. 本地搜索能覆盖工具名、`API Key`、`Base URL` 与常见报错关键词。
13. 站点结构允许后续继续新增工具页而不需要重做导航体系。

## 16. 单页完成定义

一个工具页只有同时满足以下条件，才算完成：

- 已写明实际配置入口
- 已给出最短完整步骤
- 已提供可用示例或足够明确的填写方式
- 已写出验证动作和成功现象
- 已列出高频报错及排查
- 已补必要截图或明确说明该工具无需截图

## 17. 最终设计结论

基于本次确认，首版设计结论如下：

- 站点定位：面向想尽快完成配置的用户，兼顾新手最低必要理解
- 首页策略：VitePress 默认首页，轻定制，先给 3 步快速开始，再给工具入口
- 信息架构：首页分流，`guide` 承担通用知识，`tools` 承担单页闭环
- 工具页标准：9 个工具页全部达到可直接照做的完成态，不做占位
- 平台处理：以通用步骤为主，平台差异只放注意事项
- 内容策略：先动作，后解释；先闭环，后补充
- 资源策略：不依赖外链，图片全部本地化
- 技术策略：VitePress + 纯静态部署 + 轻量搜索 + 最小定制
- 扩展策略：后续新增工具页不重做导航体系
