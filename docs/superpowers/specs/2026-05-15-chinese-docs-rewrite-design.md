# 中文文档重写设计

本次只改用户能看到的中文文档。

范围是：

- `docs/index.md`
- `docs/guide/*.md`
- `docs/tools/*.md`
- `docs/**/*.vue` 中用户能看到的中文

不改脚本提示语，不改 CI 文档，不改开发说明。

当前仓库暂时没有 `.vue` 文件。后续如果文档站组件里有中文，也按这份规则处理。

## 参考风格

参考菜鸟教程的写法：

- 首页：https://www.runoob.com/
- Git 教程：https://www.runoob.com/git/
- Docker 教程：https://www.runoob.com/docker/docker-tutorial.html

要学的是它的表达方式，不是照抄内容。

重点是：

- 标题短。
- 句子短。
- 先说结论。
- 再给步骤。
- 示例放在离步骤近的地方。
- 少写空话。
- 少写像方案书的词。

比如：

不要写：

> 本文用于帮助用户理解不同工具中的配置差异。

改成：

> 本文介绍如何填写 API Key、Base URL 和模型名称。

## 首页怎么改

首页要让用户一进来就知道三件事：

1. 这个站点是干什么的。
2. 先选自己正在用的工具。
3. 配不通时去哪里排查。

首页保留工具列表。

文字要更直接：

- 不写复杂介绍。
- 不堆接口名。
- 不把所有概念都塞在首页。
- 把 image2 单独作为图片接口入口。

## 工具页怎么改

每个工具页尽量用同一套顺序。

推荐结构：

```md
# 工具名配置教程

一句话说明这个工具是什么。

本文介绍如何在这个工具中配置 API Key、Base URL 和模型名称。

## 准备工作

## 这个工具使用什么接口

## 配置步骤

## 配置示例

## 如何验证

## 常见问题

## 注意事项
```

不是每篇都强行写满。

如果某个工具没有对应内容，就不硬凑。

## 通用指南怎么改

通用指南要像基础教程。

写法是：

1. 先说这个概念是什么。
2. 再说什么时候会用到。
3. 然后给填写规则。
4. 最后给常见错误。

比如 API Key 页面：

- API Key 是什么。
- API Key 填在哪里。
- 填写时注意什么。
- 401 报错怎么查。

Base URL 页面：

- Base URL 是什么。
- `/v1` 要不要带。
- 常见写法。
- 404 报错怎么查。

## image2 页面怎么改

image2 是图片接口，不是聊天模型配置。

这个页面要写清楚：

- 它能生成图片。
- 它能编辑图片。
- 它使用 OpenAI 图片接口格式。
- 它不适合直接填进代码助手的聊天模型设置里。

代码示例保留。

说明文字改短。

## 不做什么

本次不更新工具功能结论。

如果原文没有可靠依据，就用保守写法：

> 如果你的版本提供 Base URL 字段，可以按下面的方式填写。

不写成：

> 这个工具一定支持自定义 Base URL。

本次也不改：

- API 地址
- 字段名
- 命令
- 代码块
- 图片路径
- 外部链接
- VitePress 配置
- Vue 组件逻辑和样式

除非原文结构调整时必须移动位置。

## 写作标准

每一页改完后检查这些问题：

- 第一屏能不能看懂这页讲什么。
- 步骤能不能照着做。
- 长句有没有拆开。
- 有没有“统一、体系、链路、能力、场景化”这类空词。
- 报错说明是不是能直接排查。
- 同类页面的小标题是不是基本一致。

## 验证

重写完成后运行：

```bash
npm run docs:build
```

构建必须通过。

如果构建失败，先修正文档格式、链接或 VitePress 语法问题。
