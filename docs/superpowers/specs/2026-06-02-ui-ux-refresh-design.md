# UI/UX 改版设计说明

日期：2026-06-02

## 背景

当前站点是 VitePress 文档站，内容覆盖 API Key、Base URL、模型名称、Provider 选择和多个 AI 编程工具的配置方法。现有界面以 VitePress 默认文档样式为主，首页通过卡片列出入口，结构清楚，但视觉层级、任务引导和组件一致性仍有提升空间。

本次改版方向已确定为“清爽工具型”。目标是让站点更像可信的配置手册，而不是产品营销页。

## 目标

1. 提高首页的任务引导能力，让用户能按“创建令牌、确认 Base URL、选择工具、验证配置、排查错误”的顺序阅读。
2. 统一首页、卡片、提示区、代码块、按钮和 image2 在线工具的视觉语言。
3. 保持文档站的信息密度和可扫描性，不引入过重的装饰、动画或营销式首屏。
4. 保证中文表达准确、克制、自然，避免口语化和机械化。
5. 保持 VitePress 默认主题的可维护性，只在必要位置扩展样式和首页结构。

## 非目标

1. 不重写站点架构。
2. 不新增复杂的配置向导、登录流程或后端服务。
3. 不伪造工具截图，也不把不可验证的界面状态写入文档。
4. 不把首页改成品牌官网或广告落地页。
5. 不改变 image2 在线工具的接口行为和本地存储逻辑。

## 视觉方向

采用“清爽工具型”视觉：

- 页面基底使用白色和浅灰分区，突出文档内容本身。
- 主行动色使用稳定的蓝色，用于主按钮、重点链接和当前状态。
- 成功、检查、验证类状态使用偏青绿色，避免与主行动混淆。
- 卡片、按钮、输入框和工具面板统一使用约 8px 的圆角。
- 阴影克制，仅用于必要的层级提示，避免浮夸的玻璃拟态和大面积渐变。
- 首页可更有层次，但正文页仍以阅读体验为主。

## 首页结构

首页标题调整为：

```text
常见 AI 编程工具接入指南
```

副标题调整为：

```text
按创建令牌、填写 Base URL、选择模型和验证请求的顺序，完成工具配置并定位常见错误。
```

首页内容按以下顺序组织：

1. 顶部说明区
   - 展示标题、副标题和两个主要入口：“开始配置”和“查看故障排查”。
   - 不使用营销化口号，不使用大面积装饰图。

2. 配置前检查
   - 用简短清单展示配置前需要确认的事项。
   - 内容包括 API 令牌、Base URL、Provider、模型名称和验证请求。

3. 快速开始
   - 保留现有快速开始内容，但用步骤感更强的卡片呈现。
   - 卡片文字强调用户动作，不只解释概念。

4. 工具配置
   - 保留工具索引，按推荐路径、命令行工具、IDE 与编辑器插件、桌面客户端分组。
   - 卡片保持紧凑，便于快速扫描。

5. 能力接口
   - 保留 image2 图片接口入口。
   - 表述为能力文档，不与聊天工具配置混淆。

6. 故障排查
   - 将 401、404、模型不存在、Provider 选择错误作为明确入口。
   - 每个入口说明错误含义和下一步排查方向。

## 全站样式

样式主要落在 `docs/.vitepress/theme/style.css`：

- 定义站点级色彩变量，覆盖首页和自定义组件。
- 提升首页内容宽度，但正文页保持 VitePress 默认阅读宽度。
- 统一卡片 hover、focus、border、background 和文字层级。
- 改善提示框、代码块和表格的视觉一致性。
- 保证移动端单列布局稳定，避免按钮文字、卡片文字和表格内容重叠。

## image2 在线工具

`Image2Playground.vue` 保留现有功能：

- 输入 API Key 和 Base URL。
- 生成图片。
- 上传图片并编辑。
- 下载结果。
- 保存和清空本地历史记录。

视觉上调整为同一套工具型界面：

- 左侧为配置表单，右侧为结果和历史记录。
- 表单项分组更清晰，减少临时表单感。
- 主按钮、次按钮、模式切换和状态提示与首页视觉一致。
- 错误提示使用明确原因和下一步动作，不使用笼统失败描述。
- 移动端改为单列布局，优先显示当前操作所需信息。

## 文案标准

中文表达遵循以下规则：

1. 准确
   - 配置项名称、路径、接口名称和错误原因必须清楚。
   - 同一概念固定用词：`API Key`、`API 令牌`、`Base URL`、`Provider`、`模型名称`、`image2 图片接口`。

2. 克制
   - 不使用“轻松搞定”“一键接入”“秒级配置”等营销化表述。
   - 不对第三方工具能力作无法验证的承诺。

3. 自然
   - 避免生硬套话，例如“用户可以通过本页面进行相关操作”。
   - 避免过度口语化，例如“配不通”“填错了就看这里”。

4. 可执行
   - 尽量让每段说明指向下一步动作。
   - 错误说明应包含检查方向，而不是只描述现象。

## 可访问性和响应式

- 交互控件在移动端保持不小于 44px 的可点击高度。
- focus 状态必须可见。
- 正文、按钮和提示文字满足基本对比度要求。
- 不使用随视口宽度缩放的字体。
- 首页卡片和 image2 工具在窄屏下改为单列布局。

## 验证方式

实现完成后需要执行：

```bash
npm run docs:build
```

同时进行视觉检查：

- 首页桌面端布局无明显拥挤或错位。
- 首页移动端卡片、按钮和标题不重叠。
- image2 在线工具桌面端左右区域清楚，移动端单列可用。
- 代码块、提示框、表格和链接在亮色主题下保持可读。
